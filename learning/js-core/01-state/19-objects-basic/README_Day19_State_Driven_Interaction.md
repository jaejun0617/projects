## 📌 프로젝트 개요 (WHY)

Day 19는 객체 문법을 배우는 날이 아니다.  
이 Day의 목적은 **상태와 행동을 하나의 엔티티로 응집시키는 사고**를 만드는 데 있다.

Day 15~18에서 우리는:
- state → render 흐름
- 조건을 값으로 다루는 설계
- 이벤트 → 상태 변경
- 책임 분리 함수 설계

를 익혔다.

Day 19에서는 이 모든 개념을 **객체 하나 안에 통합**한다.

이 구조는:
- UI 컴포넌트 내부 상태 관리
- 게임/도메인 엔티티 설계
- 비즈니스 로직 모델링

의 기본 형태다.

## 🔥 오늘의 핵심 (Key Takeaways)
- 객체는 상태와 행동의 결합체다
- 상태 변경은 반드시 객체 내부에서만 일어난다
- this는 호출 주체에 따라 결정된다
- 이벤트는 객체를 직접 조작하지 않고 메서드만 호출한다

---

## 🧠 핵심 개념 / 설계 요약

### 1️⃣ 객체 = 엔티티
객체는 하나의 독립된 엔티티다.

- 상태: name, health, maxHealth
- 행동: attack(), heal(), isAlive()

UI는 객체의 상태를 **관찰만** 한다.

---

### 2️⃣ this 바인딩
메서드 내부의 this는 메서드를 호출한 객체다.

```js
hero.attack();
// this === hero
```

이 규칙이 깨지면 상태 접근도 깨진다.

---

### 3️⃣ 상태 변경 방어
- 체력은 0 미만 금지
- 최대 체력 초과 금지
- 죽은 상태에서는 행동 제한

이 방어가 없으면 UI와 상태가 쉽게 어긋난다.

---

## 🏗️ 구현 기준 & 이 과제를 통해 기른 역량

### 구현 기준
- 객체 하나 = 하나의 상태 단위
- 상태 변경은 메서드 내부에서만
- 이벤트는 객체 메서드 호출만 담당
- render는 상태를 화면에 반영만 한다

### 기른 핵심 역량
- 객체 단위 상태 관리 사고
- this 바인딩 이해
- 상태 변경 제어 감각
- 이후 다중 객체 관리(Array of Objects) 기반 완성

## ⚙️ 핵심 코드 (구조 중심)

```js
const hero = {
  health: 100,
  maxHealth: 100,

  attack() {
    this.health = Math.max(0, this.health - 15);
  },

  heal() {
    this.health = Math.min(this.maxHealth, this.health + 10);
  },
};
```
